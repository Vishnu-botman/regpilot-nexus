Design decisions to guide Phase 3 ingestion pipeline.

### 1. Regulation Lifecycle

Current schema uses a flat `status` string. Add a `regulationLifecycle` enum to distinguish:

| Status | Meaning | Ingestion Behavior |
|---|---|---|
| draft | Proposed, not yet effective | Ingest for monitoring, exclude from compliance evaluation |
| active | Currently in force | Full ingestion, embeddings, compliance evaluation |
| superseded | Replaced by newer version | Keep in DB, exclude from default compliance evaluation |
| repealed | Formally withdrawn | Keep for historical reference, exclude from all default queries |
| archived | No longer maintained but historically relevant | Keep in DB, exclude from compliance evaluation |

Separate from existing `status` field (which tracks document status: enforced/draft). `regulationLifecycle` tracks where it sits in the regulatory lifecycle.

### 2. Version Resolution

Two options for `evaluate_applicability`:

**Option A (Recommended): Latest effective version only**
- Default evaluates the latest version where `effectiveDate <= now()` and `lifecycle = active`
- Historical version evaluation available via explicit parameter
- Aligns with how regulations actually work — you comply with the current version

**Option B: All versions (current behavior)**
- Simpler, but risks duplicate obligations across versions
- May produce confusing results when v1.0 and v1.1 both have obligations for the same section

Recommendation: Option A. Add `versionResolution: 'latest' | 'all' | 'specific'` parameter. Default to `latest`.

### 3. Semantic Search Scope

**Default scope:** Search only `active` regulations with `lifecycle = active`.

**Override options:**
- `scope: 'all'` — search everything including superseded/repealed
- `regulationId: '...'` — search specific regulation (any lifecycle)
- `includeHistorical: true` — include superseded versions in results

Recommendation: Default to active-only. The `searchSimilar` query filters by `regulation.status = 'active'` unless explicitly told otherwise.

### 4. Embedding Refresh & Synchronization

Three triggers for embedding refresh:

| Trigger | Action |
|---|---|
| New regulation ingested | Generate embeddings for all new chunks |
| Version superseded | Mark old version's embeddings as stale, generate new ones for replacement |
| Regulation repealed | Soft-delete embeddings (keep for historical queries, exclude from default search) |

Implementation: Embedding generation is part of the ingestion pipeline, not a separate step. The parser produces chunks, the embedding service generates vectors, both are stored atomically. Old version chunks remain but are excluded from default search via lifecycle filter.

### 5. Retrieval Ranking

Results ranked by:

1. **Semantic similarity** (primary) — cosine distance from pgvector
2. **Applicability** (secondary) — does this regulation apply to the company?
3. **Status** (tertiary) — active > superseded > repealed
4. **Recency** (tiebreaker) — newer versions rank higher

SQL ordering:
```sql
ORDER BY
  embedding <=> $query_embedding,  -- semantic similarity
  CASE WHEN r.status = 'active' THEN 0 ELSE 1 END,  -- status
  v.effective_date DESC  -- recency
```

Start with similarity + recency. Add applicability as a post-query filter (only return chunks from regulations that match the company profile).

### Summary

| Decision | Recommendation |
|---|---|
| Lifecycle states | Add `regulationLifecycle` enum: draft, active, superseded, repealed, archived |
| Version resolution | Default to latest effective version; allow override |
| Search scope | Default to active regulations; allow scope parameter |
| Embedding refresh | Part of ingestion pipeline, not separate |
| Retrieval ranking | Similarity → status → recency; applicability as filter |
